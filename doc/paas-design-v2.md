# 内部 PaaS 平台 — 设计文档

> 评判标准:设计是否清晰、边界是否明确。实现细节(代码大部分由 AI 完成)不在此展开。

---

## 1. 定位

- **一句话**:自托管的、内部用的「基础版 Vercel + Neon」,把部署与数据资源放进同一个控制面 + CLI。
- **使用者**:公司非工程师,经由 AI(Claude Code / Codex 等)操作;也支持工程师手动用。
- **核心价值(裸服务器给不了的两点)**:
  1. **隔离** —— 一群人在同一台服务器上各自部署,彼此不踩。
  2. **可见性 / 治理** —— 管理者能看到谁在用、占多少资源,能兜底处置。
  - (「让 AI 部署」本身裸服务器也能做;平台不可替代的是隔离 + 可见性。)
- **前提**:数据不出公司 ⇒ 自托管。单台 VPS + 自管 Traefik(不用 Coolify —— 它是单租户专家工具,与「让非专家无顾虑共用」的目标不是一类东西)。

---

## 2. 核心抽象:四类资源 + 一个动词

平台只有四类资源,全部 **user 直属、平级、各自独立生命周期**:

```
user
 ├─ services    (部署单位,容器)
 ├─ databases   (Postgres 库)
 ├─ caches      (缓存 = valkey;UI 上叫「缓存」)
 └─ volumes     (文件存储)
```

唯一的核心动词:**「把资源注入给 service」**。
- database / 缓存 → 注入**连接字符串**到 service 的环境变量。
- volume → **挂载到 service 容器路径** + 注入**路径**到环境变量。

资源与 service 是**弱关联(多对多、软的)**:一个资源可注入多个 service,一个 service 可注入多个资源。删任一方不连带删另一方,只是「注入失效」。

> 这个对称让四类资源共享同一心智:**顶层独立资源 → 注入 service → AI/app 拿到一个普通接口(URL 或路径)**。AI 面对四类资源只有一种理解方式。

---

## 3. service(部署单位)

- **统一走容器**:静态站点 = nginx 容器,动态 app = 跑代码的容器。平台眼里一律「镜像 → 容器 → 域名」。
- **可以是单容器,也可以是一个 docker compose(多容器)**:用于「一个 app 的多组件共生死」(前端+后端等),compose 内部网络互通。
  - 卷:持久卷由 compose 内**指定的主容器**挂载。
  - 域名:一个 service 仍只对外一个域名 → 指向 compose 内**入口容器**;其余仅内部互通。
  - 配额:按**整个 compose 总和**计 —— 落地用共享 cgroup parent slice 在父级设上限(Docker 不原生支持多容器内存之和;备选:按容器分摊 `--memory`)。
- **一仓库 = 一 service(1:1)**:用 GitHub 仓库级权限天然隔离 push/deploy。
  - 1:1 天然对齐:GHCR token 按仓库=service;每 service 一个 deploy key = 每仓库一个,存该仓库 GitHub Secrets;建 service = 建仓库(一个动作)。
  - (避免共享仓库 + 目录前缀:目录只是命名,GitHub 不提供目录级 push 权限,挡不住改别人目录。)
- **环境变量**:是 **service 的配置**(存控制面、属于该 service 的元数据),**不落文件、不进 volume**。部署时注入容器,**先注入后启动**(否则 app 启动读不到 env 会崩)。
  - 人类手动填;AI 用户由 **CLI 自动获取资源连接串/路径写入 env**。
- **子域名**:全局唯一 `<service[-随机词]>.<域名>`(Vercel 式,冲突追加随机词)。
- IP 限制见 §8。

## 4. database(Postgres 库)

- **一键新建**:用户点「创建数据库」= 平台在**单一 Postgres 实例**里真的 `CREATE DATABASE` 一个**独立 database**(非 schema)+ 一套受限连接凭证。
- **web 操作页**:点进去能在线建表 / 索引 / 跑查询 / 看数据(平台自带轻量 web SQL 客户端),并可**查看 / 复制连接字符串**。
- **一个 database 一个连接字符串**,可 **rotate**;rotate = 旧串立即作废(预期行为,非故障)。
- **连接字符串外部也能用(像 Neon)**:
  - 连接入口 = **pooler(PgBouncer / pgcat)对公司 IP 段可达**(不是裸 Postgres);pooler 兼做连接池(性能)与对外入口。
  - **IP 白名单**:仅公司 IP 段内可连(含公司网络下的本地调试)。
  - **连接串即密码**:用户自负保管;UI 在展示处明确警示「等于数据库密码,勿泄漏 / 勿提交进 git」。泄漏可 rotate 作废。
  - **已认边界**:公司 IP 段内,任何持串者都能连该库 —— 平台不区分是不是授权的 service(换外部调试便利)。
- **隔离落法(已定死)**:单实例 + 每库独立 `CREATE DATABASE` + 独立连接凭证。
  - 用户视角只暴露「连接字符串 + 能建表」,不接触实例、看不到别人的库。
  - **接受的代价**:同实例多库共享实例资源(连接数、CPU、磁盘、崩溃半径);单库打满连接/崩实例会影响同实例其他库。用「每库连接数上限 + pooler 收敛 + 容器层资源 limit」部分缓解;实例级共享故障域是接受的代价(内部、人少、禁放真实数据)。

## 5. 缓存(valkey)

- UI 上叫「**缓存**」(对非工程师友好);底层 valkey。
- 一键新建,注入**连接字符串**到 service env(`redis://...`),app 用普通 redis 客户端。
- 隔离(v2 细化,先记录):平台记录每用户的 namespace;越界靠 **valkey ACL**(`~namespace:*` key 前缀 + 命令白名单 → 越界返回 NOPERM)强制(直连模型,平台不在数据路径上,不能靠平台逻辑挡)。
- **第一版可只做 database + volume,缓存隔离方案单独定后再加。**

---

## 6. volume(文件存储)

- **用户视角 = Windows 文件夹**:可创建 / 删除文件夹,看起来从根目录 `/` 开始 —— 实际是**用户个人目录下的路径**(逻辑假根)。
- **真实文件操作 ⇒ 必须防路径穿越(path traversal)—— 硬安全边界**:
  - 所有路径 = 「用户根目录 + 用户/AI 给的相对路径」拼接后**规范化(resolve)**,断言结果仍在用户根目录内,否则拒绝。
  - 禁止绝对路径、禁止 `..` 逃逸、不跟随符号链接。
  - 这是「假根目录」抽象的安全基础 —— 漏了这层,假根就被穿透,能读到别人/宿主机文件。
- **注入给 service**:挂载到容器路径 + 写一个**文件夹路径环境变量**(如 `STORAGE_PATH`),app 从这个路径开始读写;**目录不存在自动创建**。
- AI 用普通 `fs` / `open`,零 SDK、零理解负担(**不用 S3 那种对象存储** —— 对 AI 是额外 SDK/概念、对用户是听不懂的术语,两头更难)。
- **并发**:v1 一个 volume 默认挂一个 service(独占);共享挂载(多 service 同卷)有并发写问题,标为后做。

## 7. 认证 / 用户 / Owner

### 登录
- **Google login**(Google Cloud Console 注册 OAuth client),校验 `hd`(hosted domain)= 公司域名,非本域拒绝(公司在 Google Workspace)。**服务端校验,不信前端。**
- 登录态用 **session**。
- 后续加 **passkey**(同样限公司域名)。
- 认证**自建表手写**(沿用既有 Amber 方案);CLI 认证同样沿用 Amber。实现层,不影响架构。

### Owner(超级管理员)
- **看 / 操作两层分离**:
  - **「看」靠共享密码**:任何公司域名用户输共享密码即可进 owner 画面**查看**(只读)。owner 可随时**重置该密码**(扩散过开就重置,旧密码全失效 ⇒ 不会失控)。
  - **「操作」靠 owner 身份**:只有 owner 能执行操作。前端按身份显隐入口;**后端每个操作独立校验 owner 身份**,前端篡改无效。
- **owner 身份:最多 2 个对等 owner**(第二个可不设,补单点;两邮箱不能相同)。
  - 互相可移除,但**不能移除自己**(至少留 1 个);移除时邮件通知被移除者。
  - 冷启动:**初始 owner 由配置/env 显式种子指定**(不用 first-login-wins,避免任意人偶然先登录抢到)。

### 管理员画面(可见性 + 兜底)
- **显示用户真名 + 该用户资源的匿名编号**,平铺一起显示(看得到「谁有几个什么资源、各占多少」,看不到资源的名字和内容):

| 资源 | 显示 | 监控指标 | owner 兜底操作 |
|---|---|---|---|
| service | 用户名 + service1/2… | CPU + 内存 | 停止 / 删除 |
| database | 用户名 + db1/2… | 存储空间 | 停止 / 删除 |
| 缓存 | 用户名 + cache1/2… | 已用内存 | 停止 / 删除 |
| volume | 用户名 + vol1/2… | 占用空间 | 删除 |

- 另有:系统资源总览页、资源占用排行页(均匿名编号 + 占用量)。
- **停止仅限单个 service 容器,不含停整机**(整机是核按钮,平台自停后无法自启,不放面板)。
- **边界**:owner 看元数据/资源/用户身份,**不看内容**(库数据、文件内容、env 明文)。
  - 诚实边界:持 Postgres admin 凭证者技术上仍能读库 ⇒「不看内容」是 UI 级保证,非密码学强保证,需向团队说清。

### 高危操作二次确认
- **owner 动别人资源**(删/停别人的 service/db/缓存/volume、重置别人连接串)→ **Gmail 验证码(Resend)**。
- **用户动自己资源**(删自己的) → **输入名字确认**(GitHub 式),不发邮件,省 Resend 额度。

## 8. 构建 / 部署链路(不用 Coolify)

```
用户(+AI) → 一人一仓库 push
          → GitHub Actions: nixpacks/buildpacks build 或 AI 生成的 Dockerfile → 推 GHCR(scoped token)
          → workflow 末步回调平台「部署 API」(service id + 镜像 tag + deploy key(HMAC))
平台控制面 (React + Rust Axum) 收到回调:
          ├ 验证 deploy key(只能部署自己的 service)
          ├ docker pull → 拉起/更新容器(资源配额)
          ├ 挂载该 service 的 volume 到约定路径
          ├ 注入 env(资源连接串/路径,先注入后启动)
          └ 容器加 Traefik label → 自动路由 + Let's Encrypt
```

- 把「执行不可信用户代码的构建」推到 **GitHub Actions**(白嫖其构建隔离),平台不在宿主机跑用户构建。
- registry 用 **GHCR**(包权限可按仓库 scope)。
- **两条部署路径**(构建都不在平台宿主机):① GitHub Actions;② 本地构建(额度用尽回退,在用户自己机器构建)。
- **IP 限制(反向代理层)**:公司 IP 白名单配在 **Traefik ipAllowList**,所有 service 经入口自动继承;需公开的单独豁免。不依赖用户代码自己写。

## 9. 凭证模型(互不通用 · 各自最小权限)

| 凭证 | 给谁用 | 干什么 | 形态 / 存放 |
|---|---|---|---|
| **数据库/缓存连接字符串** | 运行中的 app / 公司 IP 内调试 | 连库 / 连缓存 | 受限凭证;注入 env;web 页可查看复制 |
| **deploy key** | CI / 本地构建脚本 | 调部署 API 部署该 service | HMAC secret;存 GitHub Secrets(仓库级) |
| **session** | 网页用户 | 维持网页 login | cookie/session |
| **CLI access token** | CLI / AI | 非交互调平台 API | 网页内签发,带用户身份与权限 |

- 四者完全独立,绝不复用。拿到部署权 ≠ 拿到连接串;拿到 CLI token ≠ 能直连别人的库。
- (HMAC 防重放配 nonce 去重,属实现层。)

## 10. CLI(让 AI 代操作)

- **目标:覆盖管理页面全部功能**,AI 可代用户操作。
- **管理页面与 CLI 都只是同一套平台 API(Axum)的前端** ⇒「CLI 覆盖全部功能」自然成立,无两套逻辑。
- 部署、查状态/日志、建资源、注入资源 —— 全走 CLI(背后调平台 API),AI 只需理解「资源」和「注入」两个概念。
- 认证用 **CLI access token**(per-user、最小权限、可吊销)。CLI 让操作简单,但**不放大权限**(token 只能在该用户范围内操作)。

## 11. 删除与回收站(软删除 · 3 天宽限 · 四条独立线)

四类资源**各自独立删除、各自独立进回收站**(解耦后无连带):

- **service**:停容器 → volume 不属于它、不连带 → 元数据进回收站。
- **database**:`pg_dump` 导出(**不压缩** —— 删除应廉价快速、不占 CPU;dump 临时多占盘,3 天后清,磁盘已告警)+ 存重建元数据(role 名 + **密码**、database 名、权限、连接串)→ DROP 库本体 → 进回收站。
- **缓存 / volume**:**直接移动到回收站区域,不压缩**(卷是现成目录,移动近乎零成本;压缩是 CPU 风险)。
- 回收站内:**直接删除**(永久)或 **复原**。
  - 复原 = 重建空壳(库:同名 role + **同密码** + 权限,故必须存密码,不能只存 dump)+ 灌回数据 / 移回卷并重挂。**复原后凭证、文件照旧,像没删过**。
- **资源被删后,注入了它的 service 会怎样**:那条 env 失效(连不上/挂不上),service 本身还在还跑 —— 和 rotate 后未重新注入是同一种情况,非特例。删资源时可轻量提示「这些 service 正注入着它」(非阻塞)。

## 12. 横切底线

- **隔离靠机制不靠纪律**:平台机制(IP 限制、受限凭证、防路径穿越、配额)是硬边界;CLAUDE.md 只指导「用户 app 内部最佳实践」,属软建议、不承担隔离职责。
- **资源管理(起步)**:
  - **内存 hard limit + CPU soft limit:第一天就开**(内存泄漏/fork bomb 是瞬时打满整机、告警来不及;AI 生成代码 bug 概率高)。内存 `--memory` 硬上限(OOM 只杀单容器);CPU `--cpu-shares` 软限。
  - **磁盘**:软上限 / 告警(磁盘满会拖垮整机)。
  - 持续高负载 → Gmail(Resend)通知 owner。
  - 精细配额数值、回收/闲置判定:等用量数据后调。
- **自动备份**(Postgres + volume):机制第一天就要;RPO 先保守默认(如每日 + 关键操作前手动),不留空,数值后调。
- **可复现部署**(IaC / docker-compose 固化宿主机 + 平台 + Traefik + Postgres):挂了能脚本重建。
- **单点故障(SPOF)—— 已认代价**:单 VPS + 单 Traefik = 全平台 SPOF,机器/Traefik 挂则全员中断。接受,靠 IaC 重建 + 备份兜恢复(内部工具,不上高可用)。
- **金融公司红线**:明文规定禁止放真实客户/交易/财务数据,平台视为内部不可信环境。
